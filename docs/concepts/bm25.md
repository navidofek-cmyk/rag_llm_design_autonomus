# BM25 — Sparse Retrieval

## What Is Sparse Retrieval?

Dense retrieval (vector search) represents a document as a single dense vector — 384 numbers, all non-zero, capturing the document's overall meaning. **Sparse retrieval** takes the opposite approach: represent a document as a very long vector where most values are zero. Each dimension corresponds to a word in the vocabulary. The vector has a non-zero value only for words that actually appear in the document.

The vocabulary of English is roughly 100,000 words, so a sparse vector is mostly zeros. This is why it is called "sparse." But the pay-off is that **exact keyword matches produce strong signals** — something dense embeddings can blur over.

Think of it this way:
- Dense retrieval asks: "Does this document talk about the *same topic* as my query?"
- Sparse retrieval asks: "Does this document contain the *same words* as my query?"

Both questions are useful. Neither alone is sufficient.

## TF-IDF: The Foundation

Before BM25, there was **TF-IDF** (Term Frequency — Inverse Document Frequency). It scores documents by two principles:

### Term Frequency (TF)

A word that appears many times in a document is more relevant to that document than a word that appears once.

```
TF(word, document) = count of word in document / total words in document
```

If "decorator" appears 10 times in a 100-word document, TF = 0.1.

### Inverse Document Frequency (IDF)

A word that appears in many documents is less informative than a rare word. "The" appears in every document — it tells you nothing. "HNSW" appears in very few documents — it tells you a lot.

```
IDF(word) = log( total_documents / documents_containing_word )
```

Common words get a low IDF. Rare words get a high IDF.

### TF-IDF Score

Multiply them together:

```
TF-IDF(word, document) = TF * IDF
```

A word scores high when it appears often in *this* document but rarely across *all* documents.

**The problem with raw TF-IDF:** Term frequency grows linearly. A document mentioning "python" 100 times gets scored 10x higher than one mentioning it 10 times, even if both documents are equally relevant.

## BM25: Better Term Frequency Saturation

**BM25** (Best Match 25 — the 25th variant explored by the researchers) improves on TF-IDF in two key ways:

### 1. Saturation

BM25 applies a saturation function to term frequency so that adding more occurrences of a word gives diminishing returns:

```
BM25_TF = tf * (k1 + 1) / (tf + k1 * (1 - b + b * doc_len / avg_doc_len))
```

Where:
- `tf` is the raw term count
- `k1 ≈ 1.5` controls saturation speed (higher = more linear)
- `b ≈ 0.75` controls length normalization strength
- `doc_len` is this document's length
- `avg_doc_len` is the average length across the corpus

The effect: going from 1 mention to 5 mentions of a keyword is a big boost. Going from 50 to 54 mentions is almost no boost. The score **saturates**.

### 2. Document Length Normalization

Longer documents naturally contain more of any given word. A 10,000-word book mentioning "recursion" twice is less about recursion than a 500-word article mentioning "recursion" twice. BM25 normalizes for document length, giving shorter, denser documents a relative boost.

### Final BM25 Score

```
BM25(document, query) = sum over query terms of:
    IDF(term) * BM25_TF(term, document)
```

## When BM25 Beats Dense Retrieval

Dense embeddings are trained to capture *semantic* similarity — they generalize well. But this generalization can be a weakness in certain cases:

**Exact technical terms.** "PEP 572 walrus operator" — the model may not have a strong embedding for "walrus operator" if it is rare. BM25 will find it precisely because the keyword appears verbatim.

**Version numbers and identifiers.** "Python 3.12 exception groups" — version numbers are poorly handled by embedding models. BM25 treats them as distinct tokens.

**Rare domain-specific jargon.** Words that appear infrequently in the embedding model's training data have less reliable embeddings. BM25 requires no training at all — if the word exists in the document, it gets scored.

**Deliberate exact-match queries.** When a user types an error message verbatim, like `"TypeError: unsupported operand type(s) for +"`, BM25 will match documents containing that exact string far better than dense search.

> **Key insight:** Dense and sparse retrieval make complementary errors. Dense retrieval fails on exact keywords; sparse retrieval fails on synonyms and paraphrases. Combining them (see [hybrid-rrf.md](hybrid-rrf.md)) captures the strengths of both.

## Why We Build the Index From Chroma

The BM25 index in this project is built by reading all documents from Chroma rather than directly from the original files. From `src/local_rag/bm25.py`:

```python
def build_index(chroma_dir: str, collection_name: str) -> BM25Index:
    client = chromadb.PersistentClient(path=chroma_dir)
    collection = client.get_collection(collection_name)
    results = collection.get(include=["documents", "metadatas"])

    documents = results.get("documents") or []
    metadatas = results.get("metadatas") or []

    if not documents:
        return BM25Index(_index=None, _chunks=[])

    tokenized = [doc.lower().split() for doc in documents]
    bm25 = BM25Okapi(tokenized)

    chunks = [
        {
            "text": doc,
            "source": meta.get("source", "") if meta else "",
            "page": meta.get("page", 0) if meta else 0,
        }
        for doc, meta in zip(documents, metadatas)
    ]

    return BM25Index(_index=bm25, _chunks=chunks)
```

There are two reasons Chroma is used as the source of truth rather than reading files directly:

1. **Single source of truth.** Chroma already has the chunked, cleaned text. Re-reading the original files would require re-doing the chunking logic, and you might get different chunk boundaries.

2. **Consistency.** The dense vectors in Chroma and the BM25 index are guaranteed to cover exactly the same set of chunks. If a chunk is in Chroma, it is in BM25, and vice versa. This is essential for hybrid retrieval to work correctly.

## Tokenization: Deliberately Simple

```python
tokenized = [doc.lower().split() for doc in documents]
```

The tokenizer is intentionally trivial: lowercase and split on whitespace. No stemming, no stopword removal, no punctuation stripping.

This works well enough for BM25 in the context of hybrid search because:
- BM25 is a secondary signal alongside dense retrieval; it does not need to be perfect.
- Simple tokenization handles code snippets better than stemming (you want `list()` and `list` to match, but a stemmer might mangle them).
- Stopword removal can accidentally drop meaningful terms like `is`, `not`, or `as` in Python code contexts.

## Searching the BM25 Index

```python
def search(index: BM25Index, query: str, top_k: int = 5) -> list[dict]:
    if index._index is None:
        return []

    tokenized_query = query.lower().split()
    scores = index._index.get_scores(tokenized_query)

    top_indices = np.argsort(scores)[::-1][:top_k]

    results = []
    for idx in top_indices:
        chunk = dict(index._chunks[idx])
        chunk["bm25_score"] = float(scores[idx])
        results.append(chunk)

    return results
```

`get_scores()` returns a score for every document in the corpus in a single NumPy array. `np.argsort(scores)[::-1][:top_k]` sorts indices from highest to lowest score and takes the top K. The `bm25_score` is attached to each result for debugging — you can see exactly how strongly BM25 matched a document.

The `if index._index is None: return []` guard handles the case of an empty collection. If no documents have been ingested yet, there is no BM25 index to search.

---

## BM25 Does Not Need Training

One subtle advantage of BM25: unlike a neural embedding model, it requires no training data, no GPU, and no downloaded weights. `BM25Okapi` reads your documents, computes term frequencies, and is immediately ready to search. This makes it:

- **Instant to update**: Add a new document, rebuild the index in milliseconds.
- **Interpretable**: You can trace exactly why a document scored highly ("it contained the word 'decorator' 5 times, which is unusual in this corpus").
- **Reliable**: No model weights to download, no version mismatches.

The `rank_bm25` package implements BM25Okapi entirely in pure Python with NumPy — no ML frameworks required.
